# Knowhow Inventory — `smb_loan_ai`

> **Scope notes:** All content here is in English. ## Project at a glance

An **agentic Business Intelligence (BI) app for small-business lending analytics**. A user types a natural-language question ("What's the default rate by industry?") into a chat UI; an AWS Bedrock-hosted LLM agent reads the database schema, writes a SQL query, executes it against the lending database, and answers in natural language with a Markdown table. The agent is built on the `strands-agents` framework and exposes three tools: get the schema, run a `SELECT`, write a debug report. Frontend is Next.js 16 + Vercel AI SDK; backend is FastAPI deployed as a Vercel Serverless Function. Database is NeonDB Postgres in production, SQLite locally. The project is a **portfolio / case-study** project demonstrating an end-to-end agentic data app — domain is SMB lending but the architecture is reusable for any "chat-with-your-DB" scenario.

## Component inventory

The inventory is grouped by area. Read top-down: items earlier in the doc are more central to understanding the system.

### 1. Singleton entry & top-level glue (`smb_loan_ai/`)

#### `smb_loan_ai/api.py`

- **What:** Re-exports the `one` singleton: `from smb_loan_ai.api import one`.
- **Why:** Gives consumers a stable, top-level import path that hides the `one/` package internals. Lets the team rename internal files freely without breaking callers.
- **Key files:** Just the re-export.
- **Depends on:** `smb_loan_ai/one/api.py`.
- **Gotchas:** Rarely needs editing.

#### `smb_loan_ai/paths.py`

- **What:** `PathEnum` class with every important file/directory path in the project (system prompt, SQLite file, schema docs, debug reports).
- **Why:** Centralized path management — no `Path(__file__).parent / "..."` scattered across files. Single source of truth when directory layout changes.
- **Key files:** `path_enum = PathEnum()` singleton at module bottom.
- **Depends on:** `pathlib`.
- **Gotchas:** `tool_write_debug_report` in `one_04_agent.py` references a `path_debug_report_md` attribute that may not be defined here — verify before relying on the debug-report tool, and add the attribute if missing.

#### `smb_loan_ai/runtime.py`

- **What:** `Runtime` singleton with `is_local()` / `is_vercel()` methods that detect environment via the `VERCEL` env var.
- **Why:** Every place that needs to branch by environment (DB engine choice, credential source) imports this single object instead of duplicating `os.environ.get("VERCEL")` checks. Adding a new deployment target (e.g., AWS Lambda) means editing one enum here, not the whole codebase.
- **Key files:** `runtime = Runtime()` singleton.
- **Depends on:** `os`, `functools.cached_property`.
- **Gotchas:** `name` is a `cached_property`, so changing `VERCEL` env var mid-process won't be picked up. That's almost always what you want, but be aware in tests.

#### `smb_loan_ai/constants.py`

- **What:** Enums and shared magic-value constants (e.g., `DbTypeEnum.SQLITE`, `DbTypeEnum.POSTGRES`).
- **Why:** Avoid magic strings/numbers scattered through the code.
- **Key files:** Various `Enum` classes.

#### `smb_loan_ai/sql_utils.py`

- **What:** Pure functions for SQL: `format_result()` (turns a SQLAlchemy result into a Markdown table), `ensure_valid_select_query()` (rejects anything that isn't a plain `SELECT`), `execute_and_print_result()` (run + format).
- **Why:** Pure functions take an `engine` argument and have no class state — easy to unit-test, easy to call from anywhere. Mirror methods on `DbMixin` thinly wrap these.
- **Key files:** `format_result`, `ensure_valid_select_query`, `execute_and_print_result`. Constants `DEFAULT_MAX_ROWS=200`, `DEFAULT_MAX_CHARS=20000`.
- **Depends on:** `sqlalchemy`, `tabulate`.
- **Gotchas:** Markdown table output was chosen (over JSON) because it's ~24% more token-efficient when the LLM reads tool output. Don't change this without measuring.

#### `smb_loan_ai/ai_sdk_adapter.py`

- **What:** Translates between the Vercel AI SDK message format used by `@ai-sdk/react` on the frontend and the AWS Bedrock Converse API format used by Strands/Bedrock on the backend.
- **Why:** The frontend uses Vercel's standard message format (parts, multimodal); Bedrock has its own `content` block format. Rather than pull in a heavy general-purpose adapter, this file does the minimum needed for this app. Also produces the SSE stream the frontend `useChat` hook expects (with reasoning blocks).
- **Key files:** `part_to_bedrock_content`, `request_body_to_agent_history`, `ai_sdk_message_with_reasoning_generator`, `get_last_user_message_text`.
- **Depends on:** `vercel-ai-sdk-mate`, `fastapi`.
- **Gotchas:** If the frontend `@ai-sdk/react` major version changes, this file usually needs updating; protocol drift is the typical breakage.

#### `smb_loan_ai/quota.py`

- **What:** Per-user monthly token quota enforcement. Records input/output tokens in DynamoDB; raises `QuotaExceededError` once the monthly cap is hit.
- **Why:** This is a public demo project — without quota, anyone could rack up the owner's Bedrock bill. Quota is a cost-safety guard.
- **Depends on:** `pynamodb`, `pynamodb-session-manager`, `boto3`.

#### `smb_loan_ai/agent_debugger.py`

- **What:** Parses a Strands `Agent` message history to separate `<thinking>…</thinking>` blocks from the final answer; used by `/api/chat` to stream "reasoning" frames before the "text" frames.
- **Why:** The frontend wants to visualize the agent's thinking; the agent's response is a single blob, so we post-parse it.
- **Key files:** `extract_text_from_messages`, `parse_response_text`.

#### `smb_loan_ai/multi_round_bedrock_runtime_chat_manager.py`

- **What:** *Legacy* multi-round chat manager from an earlier (pre-Strands) implementation.
- **Why:** Predates the Strands migration. Kept around in case the team wants to reference the old direct-Bedrock approach.
- **Gotchas:** **Not imported by the current FastAPI/Strands flow.** Treat as dead code; verify with the project owner before deleting.

### 2. Config (`smb_loan_ai/config/`)

#### `smb_loan_ai/config/conf_00_def.py`

- **What:** A `Config` dataclass with fields for AWS credentials, Bedrock model id, DB credentials. Two factory classmethods: `new_in_local_runtime()` (reads from `~/.aws/credentials` and `.env`) and `new_in_vercel_runtime()` (reads from `os.environ`).
- **Why:** Centralizes all environment-aware loading in one file. Adding a new env var = add a field + load it in both factories. The numbered prefix (`conf_00_`) signals "this is the first/primary config module" and leaves room for future config modules.
- **Key fields:** `aws_region`, `aws_access_key_id`, `aws_secret_access_key`, `model_id`, `db_host`, `db_port`, `db_user`, `db_pass`, `db_name`.
- **Depends on:** `dataclasses`, `os`, `dotenv`, `runtime.py`.
- **Gotchas:** **WHY (best guess; verify with project owner):** the `_00_` numeric prefix is a deliberate convention also used in `one/one_00_main.py` — it gives an unambiguous read order for new contributors. Treat it as a load-order signal.

### 3. The `One` class & its mixins (`smb_loan_ai/one/`)

This is the heart of the backend. `One` is a god-object-by-design — a single singleton (`one`) that has everything wired up: config, database, AWS, agent. Each capability is a separate mixin in its own file, then all composed via Python multiple inheritance.

#### `smb_loan_ai/one/one_00_main.py`

- **What:** The `One` class. Inherits `ConfigMixin`, `DbMixin`, `Boto3Mixin`, `AgentMixin`.
- **Why:** Composition over a single 1000-line class. Each mixin is independently understandable; tests for each mixin live in `tests_python/one/`. Numbered file order makes the mixin chain visible at a glance.
- **Key files:** `class One(ConfigMixin, DbMixin, Boto3Mixin, AgentMixin)`.
- **Depends on:** `one_01_config`, `one_02_db`, `one_03_boto3`, `one_04_agent`.

#### `smb_loan_ai/one/one_01_config.py`

- **What:** `ConfigMixin` — provides `self.config` (a `Config` instance) via `@cached_property`. Chooses local or Vercel factory based on `runtime.is_local()` / `runtime.is_vercel()`.
- **Why:** Lazy-loaded so `Config.new_in_local_runtime()` (which may touch disk for `.env`) doesn't run at import.
- **Depends on:** `config/conf_00_def.py`, `runtime.py`.

#### `smb_loan_ai/one/one_02_db.py`

- **What:** `DbMixin` — provides `self.local_sqlite_engine`, `self.remote_postgres_engine`, `self.engine` (auto-selected by runtime), `self.database_schema_str` (the LLM-optimized schema text), `self.execute_and_print_result(sql)`.
- **Why:** Both engines are kept around so that scripts running locally can still hit Postgres if needed (e.g., `scripts/test_run_sql_locally_cli.py`). `self.engine` is the convenience property for "just use the right one for the current runtime."
- **Depends on:** `sqlalchemy`, `paths.py`, `runtime.py`, `db_schema/api.py`, `sql_utils.py`.
- **Gotchas:** Schema reflection (`MetaData.reflect()`) is expensive — it's behind `@cached_property` so it runs once per process. If you change the DB schema, restart the server.

#### `smb_loan_ai/one/one_03_boto3.py`

- **What:** `Boto3Mixin` — provides `self.boto_ses` (a `boto3.Session`), `self.bedrock_runtime_client()` (a Bedrock Runtime client).
- **Why:** `boto-session-manager` handles credential resolution; the mixin centralizes AWS access so adding S3/DynamoDB clients later is a one-line change.
- **Depends on:** `boto3`, `boto-session-manager`.

#### `smb_loan_ai/one/one_04_agent.py`

- **What:** `AgentMixin` — provides `self.bedrock_model` (Strands `BedrockModel`), `self.agent` (Strands `Agent`), and three `@tool` methods that the agent can call.
- **Why:** This is the most-edited file. Adding a new agent capability = define a new `@tool` method and register it in the agent's `tools=[...]` list.
- **Key tools:**
  - `tool_get_database_schema()` — return `self.database_schema_str`.
  - `tool_execute_sql_query(sql)` — validate it's a `SELECT`, run it, return Markdown.
  - `tool_write_debug_report(content)` — write the agent's reasoning trace to `tmp/debug_report.md`.
- **Depends on:** `strands`, `paths.py`, `db_schema/`, `sql_utils.py`.
- **Gotchas:** The system prompt is loaded once via `@cached_property` from `prompts/bi-agent-system-prompt.md` — restart the server after editing the prompt.

#### `smb_loan_ai/one/api.py`

- **What:** Constructs `one = One()` and exports it.
- **Why:** Forces a single-instance pattern. Everyone gets the same `one`.
- **Gotchas:** Rarely modified.

### 4. Prompt & schema (`smb_loan_ai/prompts/`, `smb_loan_ai/db_schema/`)

#### `smb_loan_ai/prompts/bi-agent-system-prompt.md`

- **What:** The agent's full system prompt. Contains: tool-usage order, domain model (Customer → Application → Loan → Payments / Default_Event), 13 derived BI metrics with SQL formulas (default rate, pricing gap, approval rate, etc.), risk-grade ranges (A–E), status codes, SQLite-specific SQL patterns, response style guidelines, read-only safety constraints.
- **Why:** Keeping the prompt as a `.md` file (not a Python string) lets non-developers edit it without touching code, and lets you diff prompt changes in PRs cleanly.
- **Gotchas:** **WHY (best guess; verify with project owner):** the prompt's 13 metrics likely come from a stakeholder interview — they're domain-specific to SMB lending, not generic.

#### `smb_loan_ai/db_schema/`

- **What:** Schema extraction and LLM-optimized encoding. `extractor.py` reflects SQLAlchemy `MetaData`; `encoder.py` formats it as compact text (`*PK`, `*FK→Table.Column`, type abbreviations like `STR`/`INT`/`DEC`/`TS`/`DT`); `model.py` defines the intermediate dataclasses; `api.py` provides factory functions.
- **Why:** The compact text format is roughly 70% smaller than raw SQL DDL when fed to the LLM — significant cost savings per request because the schema is included on every agent invocation.
- **Gotchas:** Compact format is a custom convention. If a new contributor doesn't know the convention, the schema string looks cryptic. Refer them to `encoder.py` for the legend.

### 5. FastAPI entry (`api/`)

#### `api/index.py`

- **What:** The FastAPI app. Two endpoints: `GET /api/hello` (health check), `POST /api/chat` (the main chat endpoint, returns SSE stream).

## Conventions

- **Numbered file prefix (`one_00_…`, `conf_00_…`).** Signals load order and reading order. New `one/` mixins should follow the next number.
- **Singleton-per-module pattern.** `runtime`, `path_enum`, `one` are all module-level singletons. Convenient for ergonomic imports; brittle if you ever need multiple instances. There's no DI container.
- **Pure functions in `sql_utils.py` vs methods on `DbMixin`.** Pure functions take the engine as the first argument; methods use `self.engine`. Use the pure functions in scripts/tests where you want explicit control; use the methods inside the agent flow.
- **Markdown-first output for the LLM.** SQL results → Markdown tables (not JSON). Schema → compact text (not SQL DDL). Both choices are deliberate token-efficiency optimizations.
- **System prompt is a file, not a string.** `smb_loan_ai/prompts/bi-agent-system-prompt.md`. Lets non-developers edit and diff.
- **Tests live in `tests_python/` mirroring the source layout.** `smb_loan_ai/one/one_02_db.py` → `tests_python/one/test_one_one_02_db.py`. The doubled `one` in the test name (`test_one_one_02_db`) is unusual but consistent.
