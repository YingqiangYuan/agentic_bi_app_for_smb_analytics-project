# Elevation Roadmap — `smb_loan_ai`

What 3–6 more months of senior-engineer-level investment would improve. This doc is **honest**: the project is already a coherent, working agentic BI app — the gaps below are normal for a portfolio/case-study project, and most are deliberate "ship fast" tradeoffs.

---

## Executive view (priority order)

1. **Observability** — there is no structured logging, no per-request trace id, no metrics on tool calls. For a multi-step agent, this is the first thing that becomes painful in production.
2. **Test depth** — tests cover the cheap layers (`Config`, mixin wiring) but skip the layers that actually break (`/api/chat` SSE flow, agent tool invocation, SQL validation edge cases).
3. **Agent evaluation harness** — there is no eval set. Every prompt or model change is judged anecdotally. A small golden set of question→expected-SQL pairs would unlock confident iteration.
4. **Security & input safety** — `ensure_valid_select_query` is the only guard. There's no rate limiting per IP, no prompt-injection defense, no SQL-statement timeout, no result-size DoS guard beyond the row/char caps.
5. **Frontend depth** — `recharts` is in `package.json` but unused; the agent's tabular output is the demo ceiling. Adding chart visualization is a natural next feature.

---

## Upgrade areas

### 1. Observability and ops readiness

- **Current state.** Stdout logging only. `smb_loan_ai/utils.py` has a small `debug()` helper. No request id, no structured logging, no metrics, no traces. CloudWatch will see Bedrock invocations but nothing about *which user question* triggered them. Quota state lives in DynamoDB but isn't surfaced anywhere queryable.
- **Senior-engineer target state.**
  - A `request_id` generated in `api/index.py` middleware, propagated into every log line and surfaced in the SSE response.
  - Structured JSON logs (one line per log event), shipped to a sink (CloudWatch Logs Insights, or Vector → Loki).
  - Per-tool-call metrics: count, latency, error rate, token cost. A simple `with metrics.time("tool_execute_sql_query")` block in `one_04_agent.py`.
  - OpenTelemetry traces for the agent loop: span per LLM invocation, span per tool call. Visible in Jaeger/Honeycomb/Tempo.
- **Alternatives.**
  - **A. Logfire / Sentry / Datadog APM** — turnkey, great UI, costs $$. Right answer if this becomes a real product.
  - **B. OpenTelemetry + stdout → CloudWatch** — free if you're already on AWS, more setup, full data ownership.
  - **C. Strands-native observability hooks** — newer versions of `strands-agents` expose `before_tool` / `after_tool` callbacks. Cheapest first step; doesn't require a sink.
- **Knowledge prerequisites.**
  - Structured logging discipline (one event = one JSON line, top-level fields).
  - OpenTelemetry concepts: span, trace, context propagation.
  - LLM token-cost accounting (input vs output, cached input, model-specific pricing).
- **Suggested learning path.** Read the OpenTelemetry "concepts" docs → Honeycomb's "observability for LLM apps" guide → wire a single trace into `/api/chat` and visualize it in Jaeger via Docker → expand to per-tool spans.

### 2. Test depth and coverage

- **Current state.** `tests_python/` tests Config instantiation and the mixin chain. Nothing tests `/api/chat` end-to-end, the SSE stream, the Strands agent loop, or SQL validation edge cases. No fixtures for a transient SQLite test DB. No frontend tests beyond whatever's in `tests_node/`.
- **Senior-engineer target state.**
  - An integration test for `POST /api/chat` that runs against a real test SQLite, mocks Bedrock (responds with a canned tool-call sequence), and asserts the SSE stream contains the expected reasoning + answer frames.
  - Unit tests for `ensure_valid_select_query` covering edge cases (`SELECT ... ; DROP TABLE x`, comments hiding statements, CTEs, `UNION ALL`).
  - A test for `format_result` ensuring the Markdown table respects row/char caps.
  - One Playwright/Cypress smoke test that loads `/chat`, sends a message, asserts a response renders.
- **Alternatives.**
  - **A. Mock Bedrock at the boto3 layer** (using `moto` or hand-rolled stubs) — full fidelity, more maintenance.
  - **B. Mock at the Strands `BedrockModel` layer** — simpler, but couples tests to Strands internals.
  - **C. Real Bedrock in CI with a tiny cheap model** — slow, costs money, hits external dependency in CI. Usually wrong.
- **Knowledge prerequisites.**
  - `pytest` fixtures, `@pytest.mark.parametrize`, async test patterns (`pytest-asyncio`).
  - SSE response assertions (parsing the event stream synchronously in a test).
  - Why mocking the LLM at the right layer matters (otherwise tests pass on the mock but fail in prod).
- **Suggested learning path.** pytest docs → `httpx` `TestClient` patterns for FastAPI → write one happy-path SSE test, then expand to edge cases.

### 3. Agent evaluation harness

- **Current state.** Zero. Prompt or tool changes are validated by manually typing questions into the chat.
- **Senior-engineer target state.**
  - A `data/evals/` directory with 30–50 (question, expected_sql_shape_or_metric, accept_predicate) triples covering the 13 derived metrics in the system prompt.
  - A runner script: `mise run evals` runs each question through the agent, captures the generated SQL, scores it against the predicate (exact match, contains-keyword, or a small LLM-judge), and prints a pass/fail report.
  - Run on every PR; regressions block merge.
- **Alternatives.**
  - **A. Hand-rolled predicate scoring** — fast to build, brittle to refactors of the prompt.
  - **B. LLM-as-judge** — robust to phrasing, costs $$, can be flaky.
  - **C. Snapshot testing** — record the exact agent output, fail on diff. Simplest, noisiest.
- **Knowledge prerequisites.**
  - Eval design: what's a meaningful pass/fail signal for a generative system?
  - "Exact match vs semantic match" tradeoffs.
  - Eval set curation (avoid label leakage from the same dev iterating both prompt and evals).
- **Suggested learning path.** Read OpenAI's `evals` repo README → write 5 questions by hand → see how often the agent succeeds → expand to 30+ → wire into CI.

### 4. Security & input safety

- **Current state.** `sql_utils.ensure_valid_select_query` rejects anything that doesn't start with `SELECT`. Quota caps cost. There's no per-IP rate limit, no prompt-injection guard, no SQL-statement timeout, no result-size DoS check beyond the row/char caps (which the LLM is asked-not-required to respect).
- **Senior-engineer target state.**
  - Statement-level timeout in `execute_sql_query` (SQLAlchemy `engine.execution_options(statement_timeout=...)`).
  - Per-IP rate limit middleware in FastAPI (e.g., `slowapi`).
  - A prompt-injection allowlist or output filter: refuse the agent's response if it tries to call `tool_execute_sql_query` with anything other than a `SELECT`. The current validator catches this, but log+alert on attempts.
  - Hardcoded user-agent / origin checks on the chat endpoint.
  - Secret-leak guard: the agent should never echo `DB_PASS` or AWS keys; today nothing actively prevents it.
- **Alternatives.**
  - **A. Read-only DB user** — bullet-proof. The cloud DB user the app connects as should be `GRANT SELECT` only.
  - **B. Query-cost limiter** — Postgres has `statement_timeout`; SQLAlchemy can pass it through.
  - **C. WAF in front of `/api/chat`** — broad protection, overkill for a portfolio project.
- **Knowledge prerequisites.**
  - Principle of least privilege for DB users.
  - Postgres `statement_timeout`, `idle_in_transaction_session_timeout`.
  - Prompt injection patterns (OWASP LLM Top 10).
- **Suggested learning path.** Read OWASP LLM Top 10 → audit `one_04_agent.py` against each item → harden one item at a time, starting with the read-only DB user.

### 5. API / contract design

- **Current state.** `/api/chat` accepts the Vercel AI SDK request shape and returns the SDK's SSE stream. Schema is implicit (defined by `vercel-ai-sdk-mate.RequestBody`).
- **Senior-engineer target state.**
  - An explicit OpenAPI spec generated from FastAPI (already free) and committed.
  - A versioned endpoint (`/api/v1/chat`) so the contract can evolve without breaking the frontend.
  - A typed TypeScript client generated from the OpenAPI spec, used by the Next.js side instead of raw `fetch`.
- **Alternatives.**
  - **A. tRPC-style typed RPC** — strongest typing, requires a TypeScript backend or a generator.
  - **B. OpenAPI + `openapi-typescript`** — pragmatic, what most teams do.
  - **C. GraphQL** — overkill here.
- **Knowledge prerequisites.** API versioning practices; FastAPI's `app.openapi()`; `openapi-typescript` codegen.

### 6. Developer experience

- **Current state.** `mise.toml` covers most tasks. README is sparse. `CLAUDE.md` is the de-facto contributor guide.
- **Senior-engineer target state.**
  - A `CONTRIBUTING.md` (or expand `README.md`) with "how to add a new agent tool" as a five-minute tutorial.
  - Pre-commit hooks: `ruff`, `pyright`/`mypy`, `eslint`, `prettier` — gated in CI.
  - One-command setup: `mise run bootstrap` that runs `venv-create` + `inst` + `download-db` in sequence.
  - Devcontainer / nix flake for zero-friction onboarding (optional but high-impact for a portfolio).
- **Alternatives.** Pre-commit vs Husky — pick one per language; pre-commit handles both via local hooks.

### 7. Deployment maturity (CI/CD, environments, rollback)

- **Current state.** Push to Git → Vercel auto-deploys. No staging environment, no manual approval gate, no rollback procedure beyond "revert in Vercel UI".
- **Senior-engineer target state.**
  - A `staging` branch wired to a separate Vercel project pointing at a staging NeonDB branch.
  - GitHub Actions CI: `mise run test`, lint, eval suite — required for merge.
  - A `CHANGELOG.md` updated per merge.
  - Documented rollback: Vercel rollback button + DB schema migration discipline.
- **Alternatives.** Vercel-native preview deployments (one per PR) are already free — opt in.

### 8. Data layer (migrations, schema, retention)

- **Current state.** Schema is whatever exists in NeonDB; reflected at runtime. No migration tool. `tmp/data.sqlite` is downloaded from a GitHub Release, not built from migrations.
- **Senior-engineer target state.**
  - `alembic` migrations under `migrations/`. Both NeonDB and the SQLite snapshot generated from the same source.
  - A `make seed` (or `mise run seed`) command that produces a deterministic SQLite snapshot for tests and local dev.
  - Versioned schema; the LLM-optimized schema string regenerated as part of CI.
- **Alternatives.**
  - **A. Alembic** — standard Python migration tool, works with SQLAlchemy.
  - **B. Atlas / sqitch** — language-agnostic, more setup.
  - **C. Hand-written `.sql` files numbered chronologically** — simplest, doesn't scale past 10 migrations.
- **Knowledge prerequisites.** Migration safety (additive-only patterns, online vs offline migrations), seeding strategies.

---

### 9. Performance / scalability

- **Current state.** Schema reflection cached per process. Bedrock invocation is the dominant latency. No request batching. SQLite is single-writer (fine for local). Postgres connection pool is whatever SQLAlchemy defaults give.
- **Senior-engineer target state.**
  - Tune Postgres pool size for Vercel cold/warm Lambdas (a pool of 1 per Lambda instance is usually right).
  - Cache the LLM-optimized schema string at the module level (already `@cached_property`, fine).
  - Prompt caching with Bedrock — the system prompt + schema are large and reused; caching them cuts both cost and latency significantly.
  - Streaming the tool-execution phase: emit SSE frames as tool calls happen, not after.
- **Alternatives.** Bedrock prompt caching (newer Bedrock feature) is the highest-leverage win — investigate before any other perf work.
