# Interview Playbook — `smb_loan_ai`

Production-grade interviewer questions for the `/learn-this-project-interview` skill. **Model answers are reference for the skill to grade against — never read them verbatim to the candidate.**

---

## Opening — the 90-second pitch (model)

> "It's an agentic BI app for small-business lending. A user types a plain-English question into a chat UI — something like 'what's the default rate by industry?' — and an AWS Bedrock-hosted LLM agent reads the database schema, writes the SQL, runs it, and answers with a Markdown table.
>
> The frontend is Next.js 16 using `@ai-sdk/react`'s `useChat`. The backend is FastAPI deployed as a Vercel Serverless Function. The agent framework is Strands Agents, which I picked over LangChain because it's lighter and Bedrock-native. The database is NeonDB Postgres in production and a SQLite snapshot locally — selected by a single `runtime` singleton.
>
> The agent has three tools: get the schema, run a `SELECT`, write a debug report. I encoded the schema in a custom compact format (`*PK`, `*FK→Table.Column`, single-letter type codes) which is about 70% smaller than raw DDL — meaningful because the schema rides along on every request. SQL results come back as Markdown tables, which I measured at ~24% fewer tokens than equivalent JSON.
>
> Open questions I'd address next: observability is rudimentary, there's no eval harness, and the DB user isn't read-only at the grant level. Those are the three things I'd ship in the next quarter."

## Round 1 — what you have

### 1.1 "Walk me through what happens between the user pressing send and the answer appearing."

- **Model answer (3–5 sentences):** `useChat` in `components/chat/chat.tsx` POSTs the conversation to `/api/chat`. FastAPI parses the Vercel-AI-SDK-shaped body with `vercel-ai-sdk-mate`, checks the per-user quota via `smb_loan_ai/quota.py`, restores the agent's message history, and invokes the Strands `Agent`. The agent makes one or more tool calls (typically `get_database_schema` first, then `execute_sql_query`), each round-trips through Bedrock, and the response is post-processed by `agent_debugger.py` to split `<thinking>` from the answer. The two are then streamed back as SSE frames the frontend renders separately.
- **Follow-ups.** "Where in that flow could you lose data?" / "If the SSE drops mid-stream, what happens client-side?" / "Why split thinking from answer at all?"
- **Weak answer:** "It calls the API and shows the response." No mention of streaming, tool loop, or message-history reconstruction.

### 1.2 "Explain the `One` class — what is it, and why is it built this way?"

- **Model answer:** It's a singleton aggregator that composes four mixins via multiple inheritance: `ConfigMixin`, `DbMixin`, `Boto3Mixin`, `AgentMixin`. Each mixin owns one capability and lives in its own numbered file (`one_01_…` through `one_04_…`) for readability and independent testing. The benefit is composability without a thousand-line god-object; the cost is that everything implicitly shares `self`, and there's no DI container.
- **Follow-ups.** "What goes wrong if two mixins want the same attribute name?" / "Could you achieve this with plain composition instead of inheritance?"
- **Weak answer:** "It's a class with everything in it." No mention of the mixin split or its rationale.

### 1.3 "How does the agent know what's in the database?"

- **Model answer:** At startup, `DbMixin.database_schema_str` reflects the SQLAlchemy `MetaData` and encodes it via `smb_loan_ai/db_schema/encoder.py` in a compact format (`*PK`, `*FK→Table.Column`, single-letter types). The result is cached for the process. The agent's first tool call is typically `tool_get_database_schema`, which returns that string.
- **Follow-ups.** "Why custom encoding instead of raw DDL?" / "How would you handle schema evolution without restarting?"

### 1.4 "What stops the agent from running `DROP TABLE`?"

- **Model answer:** `smb_loan_ai/sql_utils.py:ensure_valid_select_query` rejects anything not starting with `SELECT`. The system prompt also explicitly forbids write operations. The next hardening step would be a read-only DB user — `GRANT SELECT` only — so even a validator bypass can't mutate data.
- **Follow-ups.** "Show me how `ensure_valid_select_query` could be tricked." / "What about prompt injection?"
- **Weak answer:** "The prompt says don't."

### 1.5 "How are local SQLite and cloud Postgres selected at runtime?"

- **Model answer:** `smb_loan_ai/runtime.py` reads the `VERCEL` env var once per process via `@cached_property`. `DbMixin.engine` returns `local_sqlite_engine` when `runtime.is_local()`, else `remote_postgres_engine`. Both engines exist as separate properties so scripts can deliberately hit one or the other.
- **Follow-ups.** "How would you add a third runtime, like AWS Lambda?" / "What breaks if the schema diverges between SQLite and Postgres?"

### 1.6 "Why FastAPI + Next.js instead of just one of them?"

- **Model answer:** The Python ecosystem owns the agent framework (Strands) and AWS SDK (boto3); the JS ecosystem owns the modern chat UI primitives (`@ai-sdk/react`, shadcn/ui). Splitting the stack lets each side use its native tools. The cost is the message-format translation layer (`ai_sdk_adapter.py`) — manageable because we only had to map two shapes.
- **Follow-ups.** "Could you do this all in Next.js with API routes?" / "Tradeoffs?"

### 1.7 "How does the system prompt teach the agent BI semantics?"

- **Model answer:** `smb_loan_ai/prompts/bi-agent-system-prompt.md` encodes the lending domain — customer→application→loan→payment lifecycle, risk grades A–E, and 13 derived metrics with SQL formulas (default rate, pricing gap, etc.). It's a Markdown file so it diffs cleanly in PRs and non-engineers can edit it. The prompt also includes SQLite-specific patterns like multiplying by `1.0` to force float division.
- **Follow-ups.** "How big is the prompt and what does that cost per request?" / "Where did the 13 metrics come from?"

## Round 2 — what you'd elevate

### 2.1 "What's missing from this project that you'd ship in the next quarter?"

- **Model answer:** Three things in priority order. (1) Observability — request ids, structured logs, per-tool spans, token-cost tracking. (2) An eval harness — 30–50 (question, expected SQL shape) pairs, scored on every PR, because today every prompt change is judged anecdotally. (3) A read-only DB user — defense-in-depth against any validator bypass. Each ships in roughly two weeks.
- **Follow-ups.** "Why those three and not, say, perf?" / "Why is observability before tests?"

### 2.2 "Walk me through how you'd add Bedrock prompt caching."

- **Model answer:** Wire it into `BedrockModel` construction in `one_04_agent.py`, marking the system prompt and the schema string as cached blocks (they're both stable across a session). Expect significant cost and latency wins because both are large and repeated. The main risk is staleness: if schema changes mid-session, cached entries reference an old shape.
- **Follow-ups.** "What's your cache-invalidation strategy?" / "How do you measure the cost win?"

### 2.3 "How would you build the eval harness concretely?"

- **Model answer:** A `data/evals/*.jsonl` with `{question, expected_sql_pattern, accept_predicate}` triples. A `mise run evals` script invokes the agent per question, captures the generated SQL, and scores it with a hybrid predicate (regex for required keywords + LLM-as-judge for semantic equivalence). Run on every PR via GitHub Actions, fail merge on regression. Start with 10 questions, grow to 50.
- **Follow-ups.** "Why hybrid scoring?" / "How do you avoid the dev who writes the prompt also writing the evals (label leakage)?"

## Round 3 — alternatives considered

### 3.1 "Why not LangChain?"

- **Model answer:** Three reasons. Strands is Bedrock-native — it's a thin shim over `boto3.converse`, which is what we want. LangChain pulls in a lot of indirection (chains, runnables, callbacks) for what's a three-tool agent. And the `@tool` decorator pattern in Strands maps 1:1 to how I think about agent tools — there's less to learn. Tradeoff: smaller ecosystem, fewer integrations.
- **Follow-ups.** "If you needed RAG, would Strands still win?" / "What about LangGraph?"

### 3.2 "Why not Bedrock Agents (the AWS console product)?"

- **Model answer:** Bedrock Agents require AWS-side configuration — action groups, function schemas, knowledge bases — that lives outside the repo. That makes PR review and version control awkward. Strands keeps the agent definition in code, which is what I want for a project I version-control.
- **Follow-ups.** "What if AWS adds better IaC for Bedrock Agents?"

### 3.3 "Why not just have the LLM return the SQL and run it server-side without a tool loop?"

- **Model answer:** Single-shot LLM-to-SQL works for simple questions but breaks when (a) the LLM needs to inspect the schema first, (b) a query needs follow-up exploration, or (c) you want the model to retry on error. The tool-loop gives all three for free. Tradeoff: each tool call is a Bedrock round-trip, so latency adds up.
- **Follow-ups.** "How many tool calls is too many?" / "Have you measured the latency breakdown?"

### 3.4 "Why custom schema encoding instead of feeding raw DDL?"

- **Model answer:** Token efficiency. Compact format is ~70% smaller than DDL; the schema rides on every request, so the savings compound. The risk is that the LLM is less familiar with my custom format than with SQL DDL — but a clear legend in the system prompt covers that.
- **Follow-ups.** "Did you A/B test correctness with vs without the compact format?" / "What if the LLM hallucinates schema details because of the abbreviation?"

### 3.5 "Why Markdown tables for SQL results, not JSON?"

- **Model answer:** Two reasons. Tokens — Markdown is ~24% leaner. And LLMs see far more Markdown tables in pre-training than JSON arrays-of-objects, so they parse them more reliably. Tradeoff: harder to programmatically post-process the output, but we don't need to here.
- **Follow-ups.** "If you wanted to add charts, would you change the output format?"

## Round 4 — problems hit (real or hypothetical)

### 4.1 "Pretend the chat hangs in production. Walk me through your debugging."

- **Model answer:** First, check whether the SSE stream is reaching the browser (devtools Network tab). If yes, the frontend is failing to render — check `useChat` state. If no, log the FastAPI process: did Bedrock return? Did the tool execution time out? Did the Postgres query hang? Add a `statement_timeout` to SQLAlchemy as a guard. Without structured logging today this is harder than it should be — which is the observability item.
- **Follow-ups.** "What's the SLA for an answer?" / "What do you do if you can't repro?"

### 4.2 "The agent answered with a number that's clearly wrong. How do you find why?"

- **Model answer:** Look at the agent's reasoning trace (the `<thinking>` block in the SSE stream, or `tmp/debug_report.md` once that's fixed). The SQL the agent ran is in there. Reproduce the SQL by hand via `scripts/test_run_sql_locally_cli.py`. If the SQL is wrong, the prompt failed to teach the right metric — fix the prompt and add an eval case. If the SQL is right but the answer is wrong, the agent mis-summarized — also a prompt issue.
- **Follow-ups.** "How would you prevent it from happening again?"

### 4.3 "What about the `tool_write_debug_report` bug you mentioned?"

- **Model answer:** Yeah — `paths.py` doesn't define `path_debug_report_md`, so any call to that tool will raise `AttributeError`. The fix is one line in `paths.py`. The deeper question is why the agent calls a broken tool: I'd reorder the prompt to make `write_debug_report` truly optional, or remove the tool until it's properly wired.
- **Follow-ups.** "How would you catch this in CI?" (Answer: an eval that exercises the tool.)

## Round 5 — production pushbacks

### 5.1 "Traffic 100×. What breaks first?"

- **Model answer:** Bedrock token spend — without prompt caching, every request re-sends the full system prompt + schema. Then DynamoDB write contention on the quota table if many users hit limits simultaneously. Then Postgres connection limits on NeonDB — solvable with pgbouncer or NeonDB's autoscaling. Fix order: prompt caching → per-IP rate limit → pool tuning. None requires re-architecture.
- **Follow-ups.** "How much would prompt caching save in dollars?" / "What if Bedrock throttles you?"

### 5.2 "A user gets the wrong account's data. P0. How do you respond?"

- **Model answer:** First confirm it's not a hallucination by reproducing the SQL. If the SQL is correct but accessed the wrong rows, the request body's user context was tampered with — audit `api/index.py` for trust assumptions. The fact that the agent has *any* access to all rows is the root problem; in a real product, the DB connection or every query would scope by tenant id, enforced at the DB level not the prompt level.
- **Follow-ups.** "How would you architect tenant isolation from scratch?"

### 5.3 "AWS Bedrock goes down for two hours. What's your story?"

- **Model answer:** Today the app is a single-region single-provider dependency on Bedrock. Mitigation: a fallback provider (`scripts/test_ai_glm.py` and `test_ai_google_gemini.py` are exploratory sketches in that direction). Real answer: multi-provider abstraction at the `BedrockModel` layer, with health-check-driven failover. Costly to build for a portfolio project, valuable for a real one.
- **Follow-ups.** "How would you test the failover without taking down Bedrock?"

### 5.4 "Someone reports the agent leaked DB credentials in its response."

- **Model answer:** That'd be a major incident. Investigate: did the credentials end up in tool output (unlikely — SQL results, not connection strings), or did the agent hallucinate them from prompt context? The connection string is in `Config` and shouldn't be reachable from the agent, but a paranoid audit would confirm no logging path emits it. Add an output filter that scans for credential-shaped strings before streaming to the user.
- **Follow-ups.** "Could the agent be tricked into reading `.env` via a tool?" (Answer: not unless we add a file-read tool — which we shouldn't.)

### 5.5 "Your dependency `vercel-ai-sdk-mate` is abandoned tomorrow. What do you do?"

- **Model answer:** It's a thin adapter — I'd inline its functionality into `ai_sdk_adapter.py`. The harder dependency is `strands-agents`; if Strands disappeared, switching to LangChain or raw `boto3.converse` is a couple-of-days port because the tool surface is small (three `@tool` methods).
- **Follow-ups.** "Why didn't you inline `vercel-ai-sdk-mate` from the start?"

## Notes for the interviewer (skill instructions)

- Always push back at least once per question. Pick the strongest follow-up listed.
- For staff/principal flavor: drop Round 4, drill harder on Round 3 and Round 5.
- For mid-level flavor: keep Round 1 and Round 4; trim Round 2 and Round 3 to one or two questions each.
- A strong candidate will volunteer tradeoffs without being asked. A weak one will state facts without explaining the choice.
- Watch for over-confidence on Round 5 — these are the questions where humility about unknowns is part of the right answer.
