# Demo Playbook — `smb_loan_ai`

How to demo this project well. **Read the "Do NOT show" section first**, then the audience tailoring matrix, then the script.

---

## Wow features (priority order)

### #1. Natural language → SQL → answer, in real time

- **What:** User types a business question in plain English. Agent fetches the schema, writes a multi-table JOIN, runs it, returns a Markdown table.
- **Why it lands:** Most demos of "AI + database" show toy queries. This one handles legitimately non-trivial BI math (default rate by industry, pricing gap by risk grade, vintage analysis by quarter).
- **What to open:** `http://localhost:3000/chat`
- **Sample question:** "What's the default rate by industry, and which industries are most concentrated in our portfolio?"

### #2. Streamed reasoning ("thinking") visible to the user

- **What:** The agent's thought process streams into the UI as it works, before the final answer arrives. User sees *how* the agent decided.
- **Why it lands:** Builds trust. Most chat UIs hide the LLM's reasoning; showing it is a differentiator.
- **What to open:** Same chat page. Ask anything; watch the reasoning block fill before the answer.

### #3. The system prompt — domain knowledge as a markdown file

- **What:** Open `smb_loan_ai/prompts/bi-agent-system-prompt.md`. It contains the 13 derived BI metrics, table relationships, and SQL patterns. The prompt *is* the product's secret sauce, in a file the project owner can read and edit.
- **Why it lands:** It surfaces the "knowledge engineering" effort. Reviewers who care about prompt design will notice this; reviewers who don't will at least see it's intentional, not vibes.
- **What to open:** `smb_loan_ai/prompts/bi-agent-system-prompt.md` in your editor or as raw text in GitHub.

## Recommended sequence — 5-minute version

Use for: hiring managers, low-attention audiences, audience-(b) (unrelated role).

1. **(0:00–0:30) One-sentence pitch.**
   > "This is an AI agent that lets a non-technical operator ask plain-English questions about a lending database, and answers them by writing and executing SQL itself."
2. **(0:30–1:30) Show the chat answering one question.** Open `/chat`. Type *"How many customers do we have by industry?"*. Watch the streaming reasoning, then the table.
3. **(1:30–3:00) Ask a harder one.** *"What's the default rate by industry, and which industries are over-concentrated?"* — watch a multi-table JOIN appear in the reasoning and a richer table come back.
4. **(3:00–4:00) Open the system prompt** (`smb_loan_ai/prompts/bi-agent-system-prompt.md`) for ~30 seconds. Say: "This is the agent's domain brain — 13 lending metrics with SQL formulas, plus the table relationships."
5. **(4:00–5:00) Close.** "Stack: Next.js front, FastAPI back, Strands agent framework over AWS Bedrock, Postgres in cloud / SQLite locally. Deployed on Vercel. Happy to dig into any of those."

## Recommended sequence — 15-minute version

Use for: peer engineer, technical hiring manager, audience-(a) (related role).

1. **(0:00–1:00) Pitch + stack.** Same one-sentence framing, plus a 30-second stack tour.
2. **(1:00–3:00) Run the chat live.** Two questions, easy then hard, narrating what the agent is doing.
3. **(3:00–6:00) Architecture walkthrough.** Open `01-knowhow-inventory.md`'s architecture diagram and walk through: frontend → FastAPI → Strands agent → Bedrock → tool calls → DB → SSE back. Stop at the `One`/Mixin pattern and explain why.
4. **(6:00–9:00) Show the tools.** Open `smb_loan_ai/one/one_04_agent.py`. Walk through the three `@tool` methods. Show how the system prompt instructs the agent to call them in order.
5. **(9:00–11:00) Show the prompt.** Walk through `bi-agent-system-prompt.md` — the metrics, the JOIN patterns, the safety constraint (read-only).
6. **(11:00–13:00) Discuss tradeoffs.** Pick two from `03-elevation-roadmap.md` — usually "observability" and "evals". Frame as "next 3 months".
7. **(13:00–15:00) Q&A buffer / handle their interruptions.**

## Recovery moves — if something fails live

| Failure | Recovery |
|---|---|
| Chat hangs after sending a message | "Sometimes Bedrock cold-starts. Let me restart" — `mise run kill && mise run dev`. While it boots, narrate the architecture. |
| Agent answers without calling tools (random hallucination) | Acknowledge it: "That can happen when the prompt is in a weird state — let me reload." Restart the FastAPI process. |
| `tool_write_debug_report` errors | Skip discussing the debug-report tool. Stick to the two working tools. |
| Database connection fails | Pivot to opening the SQLite file in any GUI (DBeaver/TablePlus) and showing the schema, narrate what the agent *would* do. |
| Quota exceeded mid-demo | Open a backup tab to a pre-recorded answer screenshot, or pivot to walking through the code. Don't grovel; just move on. |

## Rehearsal questions (the audience may ask)

Use these to drill. The answer should fit in 30 seconds.

1. **"How do you stop the agent from running `DROP TABLE`?"**
   → Read-only validator in `sql_utils.ensure_valid_select_query`. Plus the DB user can be `GRANT SELECT` only (this is the next hardening step). The system prompt also explicitly forbids it.
2. **"Why Strands instead of LangChain?"**
   → Lighter, Bedrock-native, no LangChain abstraction overhead. We exposed three tools; LangChain would have been ~3× the lines for the same.
3. **"What's the cost per query?"**
   → Mostly the Bedrock input tokens (schema + prompt + history). Output is small. The Markdown-table + compact-schema-string choices were both token-efficiency optimizations.
4. **"What if the question needs a chart, not a table?"**
   → That's the next feature. `recharts` is pre-installed for it. Today, we'd answer with the table and the user reads the numbers.
5. **"How would you scale this to 1000 users?"**
   → Bedrock prompt caching for the system prompt + schema, per-IP rate limiting, Postgres read replica via NeonDB branching, and a separate quota table. See the elevation roadmap.
